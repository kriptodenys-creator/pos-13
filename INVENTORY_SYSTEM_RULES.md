# Правила роботи системи обліку складу та рецептур

## 📦 Базові принципи

### 1. Базові одиниці зберігання

Система завжди зберігає товари на складі в **базових одиницях**:

- **Вагові товари** → грами (g) або мілілітри (ml)
- **Штучні товари** → штуки (pcs)

### 2. Прихід товару

При приході товар може надходити в **будь-яких одиницях**:
- штуки (pcs)
- коробки (pack)
- кілограми (kg)
- літри (l)

**Система автоматично конвертує все в базові одиниці:**

```
Приклад 1: Вагові товари
Прихід: 2 коробки м'яса
1 коробка = 5 кг
→ Система зберігає: 10 кг = 10,000 г

Приклад 2: Штучні товари
Прихід: 3 упаковки яєць
1 упаковка = 10 шт
→ Система зберігає: 30 шт
```

### 3. Списання за рецептами

**Рецепти завжди списують в базових одиницях:**
- Вагові інгредієнти → грами (g)
- Штучні інгредієнти → штуки (pcs)

```
Приклад:
Склад: 10,000 г м'яса
Рецепт: 150 г на порцію
Замовлення: 1 порція
→ Списується: 150 г
→ Залишок: 9,850 г
```

### 4. Конвертація одиниць

#### Вагові товари:
- 1 кг = 1,000 г
- 1 л = 1,000 мл

#### Штучні товари:
- Зберігаються "як є" в штуках
- Конвертація не потрібна

### 5. Захист від недостатнього залишку

**Система НЕ дозволяє списання, якщо товару недостатньо:**

```
Склад: 100 г борошна
Рецепт потребує: 150 г
→ ⚠️ СПИСАННЯ НЕ ВИКОНУЄТЬСЯ
→ Виводиться попередження в консоль
```

## 🔄 Потік даних

```
1. ПРИХІД ТОВАРУ
   ↓
   Введення: 2 коробки × 5 кг = 10 кг
   ↓
   Конвертація: 10 кг × 1000 = 10,000 г
   ↓
   Збереження в БД: current_stock = 10,000 г

2. СПИСАННЯ ЗА РЕЦЕПТОМ
   ↓
   Рецепт: 150 г на порцію
   ↓
   Перевірка: 10,000 г >= 150 г ✅
   ↓
   Списання: current_stock = 10,000 - 150 = 9,850 г
   ↓
   Запис руху: inventory_movements (type='out', quantity=150)
```

## 📊 Відображення залишків

**Залишки ЗАВЖДИ відображаються в базових одиницях:**

```
Склад:
- М'ясо: 9,850 г
- Яйця: 30 шт
- Молоко: 5,000 мл
```

## ⚙️ Технічна реалізація

### Структура БД

```sql
-- Таблиця товарів
CREATE TABLE inventory_items (
  id INTEGER PRIMARY KEY,
  name_uk TEXT,
  name_lt TEXT,
  unit TEXT,              -- Базова одиниця: 'g', 'pcs', 'ml'
  current_stock REAL,     -- Завжди в базових одиницях
  cost_per_unit REAL,
  updated_at TIMESTAMP
);

-- Таблиця рухів
CREATE TABLE inventory_movements (
  id INTEGER PRIMARY KEY,
  inventory_item_id INTEGER,
  movement_type TEXT,     -- 'in' або 'out'
  quantity REAL,          -- Завжди в базових одиницях
  reason TEXT,
  created_at TIMESTAMP
);

-- Таблиця рецептів
CREATE TABLE recipe_ingredients (
  id INTEGER PRIMARY KEY,
  recipe_id INTEGER,
  inventory_item_id INTEGER,
  quantity REAL,          -- Кількість в одиницях рецепту
  recipe_unit TEXT        -- Одиниця в рецепті: 'g', 'kg', 'pcs'
);
```

### Логіка конвертації (код)

```typescript
// Визначаємо тип товару
const isWeightItem = stockUnit === 'g' || stockUnit === 'kg' || 
                     stockUnit === 'l' || stockUnit === 'ml'
const isPieceItem = stockUnit === 'pcs' || stockUnit === 'pack'

if (isWeightItem) {
  // Конвертуємо вагу в грами
  if (recipeUnit === 'kg') {
    quantityInBaseUnits = recipeQuantity * 1000  // кг → г
  } else if (recipeUnit === 'g') {
    quantityInBaseUnits = recipeQuantity         // вже в грамах
  }
} else if (isPieceItem) {
  // Штуки залишаються як є
  quantityInBaseUnits = recipeQuantity
}

// Перевірка перед списанням
if (availableStock >= quantityInBaseUnits) {
  // Списуємо
  updateInventoryStock.run(quantityInBaseUnits, itemId)
} else {
  // Попередження - списання НЕ виконується
  console.warn('⚠️ НЕДОСТАТНЬО НА СКЛАДІ')
}
```

## 🎯 Переваги системи

1. ✅ **Точність обліку** - немає розбіжностей між складом і рецептами
2. ✅ **Прозорість** - всі залишки в одних одиницях
3. ✅ **Захист** - неможливо списати більше, ніж є
4. ✅ **Гнучкість** - прихід в будь-яких одиницях
5. ✅ **Простота** - рецепти завжди в грамах/штуках

## 📝 Приклади використання

### Приклад 1: М'ясо (вагове)

```
ПРИХІД:
- Надходить: 2 коробки
- 1 коробка = 5 кг
- Система зберігає: 10,000 г

РЕЦЕПТ:
- Бургер потребує: 150 г м'яса
- Замовлення: 5 бургерів
- Списується: 150 × 5 = 750 г

ЗАЛИШОК:
- Було: 10,000 г
- Списано: 750 г
- Залишилось: 9,250 г
```

### Приклад 2: Яйця (штучне)

```
ПРИХІД:
- Надходить: 3 упаковки
- 1 упаковка = 10 шт
- Система зберігає: 30 шт

РЕЦЕПТ:
- Омлет потребує: 3 шт яєць
- Замовлення: 2 омлети
- Списується: 3 × 2 = 6 шт

ЗАЛИШОК:
- Було: 30 шт
- Списано: 6 шт
- Залишилось: 24 шт
```

### Приклад 3: Недостатній залишок

```
СКЛАД:
- Борошно: 100 г

ЗАМОВЛЕННЯ:
- Піца потребує: 150 г борошна

РЕЗУЛЬТАТ:
- ⚠️ СПИСАННЯ НЕ ВИКОНАНО
- Консоль: "НЕДОСТАТНЬО НА СКЛАДІ: потрібно 150 г, доступно 100 г"
- Замовлення створюється, але інгредієнт не списується
```

## 🔧 Налаштування

### Додавання нового товару

```sql
INSERT INTO inventory_items (name_uk, unit, current_stock)
VALUES ('М''ясо яловиче', 'g', 0);
-- unit='g' означає, що товар зберігається в грамах
```

### Створення рецепту

```sql
INSERT INTO recipe_ingredients (recipe_id, inventory_item_id, quantity, recipe_unit)
VALUES (1, 5, 150, 'g');
-- quantity=150, recipe_unit='g' означає 150 грамів
```

## 📞 Підтримка

При виникненні проблем перевірте:
1. Чи правильно вказана базова одиниця товару (`unit` в `inventory_items`)
2. Чи достатньо товару на складі (`current_stock`)
3. Чи правильно вказані одиниці в рецепті (`recipe_unit`)
4. Перегляньте консоль для детальних логів списання
